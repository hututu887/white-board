## 1. 实时协作与网络（Socket.IO）

### Q1：并发冲突处理——两个用户同时移动同一个图形时怎么处理？了解 CRDT/OT 吗？

**我的核心策略是：服务端权威 + 乐观更新 + 版本校验 + 冲突降级。**

1. **数据模型上避免“盲写”**
   - 每个图形都有 `shapeId`、`version`、`updatedAt`、`updatedBy`。
   - 客户端发操作时带上 `baseVersion`（我编辑时看到的版本）和 `patch`（比如 dx/dy 或新坐标）。
   - 服务端只接受 `baseVersion === currentVersion` 的操作，成功后版本号 +1，再广播最终状态。

2. **同时编辑同一对象时的规则**
   - 我们使用 **对象级最后提交生效（LWW）+ 版本门禁**，不是全局 LWW。
   - A、B 同时拖动同一个图形：
     - 先到服务端且版本匹配的请求被接收；
     - 后到请求如果 `baseVersion` 过期，服务端拒绝并回传最新对象；
     - 客户端收到 reject 后执行“重放本地意图”或提示冲突（根据交互场景选择）。

3. **为什么没有直接上 CRDT/OT**
   - **OT** 更适合线性文本编辑（如文档字符流），图形白板是对象图结构，OT 适配成本较高。
   - **CRDT** 很适合离线协作与最终一致，但会引入元数据膨胀、GC、合并复杂度。
   - 我当前是在线协作优先场景，所以先用“版本化操作日志 + 服务端仲裁”拿到可控复杂度；后续如果离线需求增强，会优先评估 **Yjs/Automerge + 对象 CRDT 映射**。

4. **可扩展演进路径**
   - 当前：对象级冲突仲裁（简单、稳定、易调试）。
   - 下一步：给“拖拽”引入短时软锁（如 500ms 心跳续约），降低同对象竞争概率。
   - 更远期：关键对象上 CRDT 化，普通对象维持版本仲裁，做分层一致性策略。

**面试可加分一句：**
“我不是只会说 CRDT 概念，而是根据业务在线率、复杂度预算、可观测性需求做了分阶段一致性设计。”

### Q2：广播风暴与降频——高频拖拽下如何平衡实时性与服务端压力？防抖策略怎么写？

**我的原则是把事件分层：高频预览走节流，低频落库走防抖。**

1. **事件分层**
   - `drag-move`（高频）：用于协作者看到“正在移动”，可丢包，允许近似。
   - `drag-end`（低频）：最终坐标，必须可靠，写入快照/操作日志。
   - 好处：把“视觉连续性”和“数据正确性”拆开处理。

2. **前端发送策略**
   - 鼠标移动事件本身可能是 100~200Hz，我会先在前端用 `requestAnimationFrame` 对齐到帧，再做 16~33ms 节流发送。
   - 发送内容尽量小：`{shapeId, x, y, versionHint}` 或 `{dx, dy}`，不发整画布。
   - 对同一图形在一个时间窗内只保留最后一个位置（coalescing 合并）。

3. **服务端广播策略**
   - `socket.broadcast.to(room)` 跳过回环，发给别人不回发给自己。
   - 房间级限频：同对象同用户在窗口内仅广播最新状态。
   - 弱网用户降级：可根据客户端 RTT 标记为低频通道（比如 50ms 一次）。

4. **防抖快照策略**
   - 拖拽中不频繁持久化，拖拽结束后或静默 300ms 再写快照。
   - 连续编辑时重置定时器，避免“每次 move 都落库”。
   - 关键节点仍强制保存：离开页面、切房间、手动保存。

5. **如何证明平衡有效**
   - 指标：消息量 QPS、单房间广播峰值、P95 延迟、CPU 占用、丢包感知率。
   - 目标：用户视觉上连续（>30fps 体感）同时服务端稳定不过载。

### Q3：状态错乱处理——时间戳校验如何避免延迟包导致状态回退（Flicker）？

**只靠时间戳不够，我用“单调序列 + 版本门禁 + 幂等应用”三件套。**

1. **为什么仅时间戳有风险**
   - 客户端时钟不完全可信，跨设备会有偏差。
   - 网络抖动会导致旧包晚到，单看到达顺序不可靠。

2. **正确做法**
   - 服务端为每个对象分配单调递增 `version`（或 `opSeq`）。
   - 客户端维护 `lastAppliedVersion[shapeId]`。
   - 收到更新时：
     - 若 `incomingVersion <= lastAppliedVersion`，直接丢弃；
     - 否则应用并更新 `lastAppliedVersion`。

3. **处理“我本地刚拖完，又收到旧广播”的闪烁**
   - 本地乐观更新会立刻改 UI；
   - 若后续收到低版本包，不会覆盖当前状态；
   - 若收到更高版本但位置变化大，可做 80~120ms 插值过渡，避免瞬间跳变。

4. **乱序与丢包补偿**
   - 检测到版本跳跃（比如从 10 收到 13），先暂存并请求补包或拉一次对象快照。
   - 无法补齐时走“服务端权威快照覆盖”，保证最终一致。

5. **一句话总结**
   - “避免回退不是靠延迟小，而是靠协议层保证状态单调前进。”

## 2. 画布性能优化（SVG & Zustand）

### Q4：重渲染控制——如何避免一个坐标变动导致全量图形重绘？

**关键在于 store 结构设计 + selector 粒度 + 拖拽临时态分离。**

1. **状态切片**
   - 用 `shapesById`（对象字典）替代大数组，单个图形更新只替换对应 key。
   - UI 状态（选中、hover、工具态）与业务图形状态分仓，避免互相连带渲染。

2. **细粒度订阅**
   - 每个图形组件只订阅 `state.shapesById[id]`。
   - 搭配 `shallow` 或自定义比较器，确保引用不变就不触发重渲染。
   - 列表层只订阅 id 列表，不订阅每个图形内容。

3. **拖拽期优化**
   - 高频位移先写入 `ref` 或临时层（如 transform），不立即写全局 store。
   - `drag-end` 再一次性提交最终值到 store，减少 React commit 次数。

4. **组件层优化**
   - 图形组件 `React.memo`；
   - 复杂路径缓存 path 计算结果；
   - 非可见图层可 `display:none` 或卸载。

5. **验证方式**
   - 用 React Profiler 看 commit flamegraph；
   - 观察拖拽时 commit 次数、单帧耗时；
   - 指标目标：大量图形下交互仍稳定在可接受帧率。

### Q5：SVG vs Canvas——为什么选 SVG？1000+ 图形怎么处理性能瓶颈？

**选择 SVG 是因为“编辑器语义友好”，而不是因为它绝对更快。**

1. **为什么首选 SVG**
   - 每个图形是天然 DOM 节点，事件命中、样式控制、选中框、无障碍和导出都更直接。
   - 对白板编辑器而言，开发效率和可维护性非常关键。

2. **SVG 的瓶颈**
   - 节点过多（1000+）时，DOM 树膨胀、样式计算和重排成本上升。
   - 频繁修改属性会触发主线程压力。

3. **我的优化手段**
   - **分层（Layers）**：背景网格、静态图层、活动图层分离，减少全局更新影响。
   - **可视区裁剪**：只渲染 viewport 内或附近对象（虚拟化思想）。
   - **降采样**：缩放比例很小时用简化图元（例如复杂路径转包围框预览）。
   - **交互态与静态态分离**：正在拖拽的对象放到独立 overlay 层。

4. **何时考虑 Canvas/Hybrid**
   - 当对象数量和动画频率继续上升，且“节点语义”收益下降时，会转为 **SVG+Canvas 混合架构**：
     - 可编辑/可选中对象留在 SVG；
     - 海量静态背景对象绘制到 Canvas。

5. **面试中的工程判断**
   - 我会强调：技术选型不是教条，而是根据“可维护性、交互复杂度、性能边界”动态演进。

### Q6：坐标转换——平移缩放下如何做 viewport 到画布坐标映射？

**本质是统一变换矩阵，所有命中和绘制都走同一套坐标系。**

1. **维护视图状态**
   - 维护 `panX/panY/scale`（或完整 3x3 矩阵）。
   - 鼠标事件拿到的是屏幕坐标 `screenX/screenY`。

2. **逆变换求画布坐标**
   - `world = inverse(ViewMatrix) * screen`。
   - 常见简化公式：
     - `worldX = (screenX - panX) / scale`
     - `worldY = (screenY - panY) / scale`
   - 所有图形数据存 world 坐标，避免缩放后数据污染。

3. **以鼠标点为中心缩放**
   - 缩放前先算鼠标所在 world 点；
   - 更新 scale 后反推 pan，使该 world 点映射回同一屏幕位置，避免“缩放漂移”。

4. **命中检测一致性**
   - 点击命中时先把鼠标点转换到 world 再做几何判断；
   - 不能直接拿屏幕坐标和图形 world 数据比较，否则缩放后误差大。

5. **工程注意点**
   - 限制 scale 上下界；
   - 使用浮点容差 epsilon；
   - 连续缩放后可定期归一化矩阵，避免误差累积。

## 3. 多人编辑模型

### Q7：撤销/重做——A 删除图形后，B 撤销该图形移动会怎样？如何解耦跨用户依赖？

**我的原则是：撤销的是“我的意图”，不是强行回滚全局历史。**

1. **撤销栈按用户分离**
   - 每个用户维护自己的 `undo/redo` 栈，记录操作及其目标对象版本。
   - 这样不会让一个用户的 Ctrl+Z 直接回滚其他人的完整历史。

2. **执行撤销前做前置校验**
   - 校验对象是否存在；
   - 校验当前版本是否满足该逆操作前提。
   - 若前提不成立（例如对象已被他人删除），该撤销项标记为失效并跳过。

3. **冲突下的降级策略**
   - 软失败：提示“目标对象已变化，已跳过本次撤销”；
   - 或补偿：把“撤销移动”转成“恢复到最近可用状态的重建/重定位”。

4. **为什么这样设计**
   - 避免跨用户强耦合引发不可预期连锁回滚；
   - 让撤销语义对操作者可解释、可预期。

5. **进一步优化方向**
   - 操作依赖图（DAG）记录前后置关系；
   - 对强依赖操作做事务化批处理撤销。

### Q8：权限控制——只读角色是在前端拦截还是服务端校验？高并发如何保证性能？

**必须双端做，但最终以服务端校验为准。**

1. **前端拦截的作用**
   - 快速反馈（禁用按钮、隐藏编辑手柄），优化体验。
   - 但前端永远不可信，只能算 UX 层。

2. **服务端强校验**
   - 每个 socket 连接在握手后绑定 `userId/roomRole`。
   - 每条写操作事件先过权限中间件：无写权限直接 reject，不进入业务逻辑。

3. **高并发性能做法**
   - 权限信息放内存/Redis 热缓存，避免每条事件查数据库。
   - 角色变更走发布订阅推送到网关节点，保持缓存新鲜。
   - 事件鉴权尽量 O(1)：`roomId + userId -> role`。

4. **安全细节**
   - 事件参数做 schema 校验，防止越权字段注入。
   - 审计日志记录拒绝事件与异常行为，便于风控。

5. **一句话总结**
   - “前端负责友好，后端负责正确；高并发下靠缓存与中间件链路保证吞吐。”

## 4. 导出与工程化

### Q9：SVG 导出 PNG——外链图片和 CSS 样式如何处理跨域与样式丢失？

**导出的关键是“资源本地化 + 样式内联化 + 安全降级”。**

1. **样式丢失原因**
   - 仅序列化 SVG 标签时，外部 CSS 规则不会自动带入；
   - 浏览器在 canvas 绘制时不认识原页面的级联上下文。

2. **我的处理方式**
   - 导出前遍历 SVG 节点，把 `getComputedStyle` 关键属性内联到元素；
   - 生成“自包含 SVG 字符串”后再转 Blob/URL 绘制到 canvas。

3. **外链图片跨域问题**
   - 图片若无 CORS 头，绘制后 canvas 会 tainted，无法 `toDataURL`。
   - 方案：
     - 优先同源资源；
     - 或通过后端代理拉取并转发 CORS；
     - 或在导出前转 dataURL（受体积限制）。

4. **字体与滤镜**
   - 字体可预加载并在导出前确认 `document.fonts.ready`；
   - 复杂滤镜在不同浏览器表现不一致时提供兼容降级样式。

5. **工程实践**
   - 提供导出前检查清单：跨域图片数量、未内联样式节点、潜在失败原因；
   - 导出失败给用户可理解提示，而不是静默失败。

### Q10：选区与碰撞检测——如何判断命中？旋转图形如何检测？

**我采用“先粗后精”的两阶段命中检测。**

1. **第一阶段：粗筛**
   - 用 AABB（轴对齐包围盒）快速排除绝大多数不可能命中的图形。
   - 可结合空间索引（四叉树/R-Tree）把查询从 O(n) 降到近似 O(log n)。

2. **第二阶段：精筛**
   - 对候选图形做精确几何命中：
     - 矩形/椭圆：解析式判断；
     - 路径：point-in-path 或射线法。
   - 对旋转图形，先把点击点通过图形变换矩阵的逆矩阵变换到局部坐标，再按未旋转形状判断。

3. **为什么不用“纯像素检测”**
   - 像素级检测准确但成本高、实现复杂、缩放下易受抗锯齿影响。
   - 几何法在编辑器场景更可控、可解释、可优化。

4. **框选与多选**
   - 框选用“选择框与图形包围盒相交/包含”策略；
   - 对旋转图形可进一步用 SAT（分离轴定理）提升精度。

5. **交互细节优化**
   - 设定命中容差（如 4~8 像素）提升细线可选性；
   - 处理 z-index：从顶层向下检测，命中即停；
   - 支持锁定图层跳过命中，避免误选。

## 面试表达模板（建议背熟）

你可以按这个模板回答每一道追问：

1. **先给结论**：我采用了什么策略，目标是什么。  
2. **再讲机制**：协议/数据结构/核心算法怎么设计。  
3. **再讲取舍**：为什么不用另一个方案（如 CRDT/Canvas 全量）。  
4. **最后讲结果**：性能指标、稳定性收益、线上问题如何下降。  

这样讲法会让面试官感受到你不仅“做过”，而且“有架构判断和工程复盘能力”。
